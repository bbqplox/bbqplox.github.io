WhoseChore

WhoseChore is a web application developed using Ruby on Rails framework and PostgreSQL. It helps housemates manage chore responsibilities through the application's chore rotation feature. Coupled with email reminders, getting tasks done has never been easier.

Check it out.

WhoseChore was developed using agile software development principals with a team of 10 members to serve the customer(our advisor). My role during the project was being the software architect. During the coding sprints, my main contributions for the project was implementing the main functionality of the web application and its interaction with the database.

Week 1 Brainstorming

The team spent the first week pitching ideas, searching for a problem to solve. In the beginning we mostly pitched broad scoped ideas that did not seem realistic to implement within a small stretch. We thought of things ranging from creating a website that provides learning resources for UCSD courses to mobile apps that help college students move out of their apartments. In the end, we settled on an idea that we all had problems with: micromanaging who cleans what in a home shared between college students.

Week 2-3 Need finding

The following two weeks were spent getting to know our potential users. We sent surveys through social media and interviewed individuals throughout campus with open ended questions to avoid biases in the features of our application. We then compiled the esults and made user stories for each of the queries the users want addressed. Then we came up with use cases to help determine the requirements of our application.

We started to think about what kind of platform our app should be implemented on and made some wire-frames to help us decide. The wireframes were made for a mobile app, but we decided to make a web app since it would be more convenient for the users to ensure that all the housemates have the ability to use the app. Thus, we decided on making a web application.

Weeks 4-5 Requirements

For the next iteration of developing WhoseChore, the next artifact we created was the use case diagram, which helps us visualize the workflow of our end product. We also filtered the most important use cases to make our app function and created a list of system requirements. This helped us sort our priorities in when we implemented the application in following weeks. This lead us to creating design use cases, which describe the exact workflow that the user has to go through to use one of the features in the application.

Weeks 6-8 Planning for the sprints

Even when we are more than half way through the development processes, we still have not began coding at this point. More planning is needed before implementation. We created sample screens to help us visualize the UI for our website and a database schema so that we will have a well planned out structure to store the user information. The schema we made was based off of an SQL database, which came out much useful later.

One of the more difficult artifacts we came across is the system test plan. This document involved a meticulous step-by-step testing process for every one of the features for our application. We had to visualize that we had the application done in order to write this out.

Weeks 9-10 The sprint

We decided to implement the application with Ruby on Rails using PostgreSQL and using Heroku to host our website. We thought that Rails would make it easier for us to manage an MVC software architectural pattern. I stored and queried user and group information using Ruby as the server side language in order to implement the user's chore list and groups of users. I also set up permissions so that all types of users were restricted to only view what they were supposed to on the web page. In the end, my team did a great job implementing the electronic mailer and chore rotation algorithm that acted as the core base of the application, as well as managing invites and logging in.
